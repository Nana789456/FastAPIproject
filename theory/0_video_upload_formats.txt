Загрузка видео по API: теория и 3 основных варианта

Что такое form в HTTP-запросе
- Form - это способ передать поля как набор именованных значений, а не как JSON.
- Для файлов обычно используют multipart/form-data.
- В этом формате в одном запросе можно передать текстовые поля и сам файл.

Можно ли без form
- Да, но тогда нужно выбрать другой способ передачи файла.
- Для обычного API с файлами multipart/form-data - самый практичный вариант.
- Если убрать form, остаются варианты:
  - передавать файл отдельно как бинарное тело запроса;
  - кодировать файл в base64 и класть в JSON;
  - сначала получить ссылку для загрузки и отправлять файл туда.

Нужно ли прикреплять файл руками
- Нет, не нужно.
- Клиент API сам отправляет файл в запросе.
- "Прикрепить файл" означает только то, что клиент кладёт байты файла в HTTP-запрос.

Как это выглядит по смыслу
- JSON: "вот данные"
- multipart: "вот данные и вот файл"
- raw binary: "вот только файл"

1. JSON
- В запросе передаются обычные данные в JSON.
- Файл напрямую сюда обычно не кладут.
- Можно положить файл в base64, но это неудобно.

Пример:
{
  "title": "Урок 1",
  "author": "Нана",
  "course": "Курс 1",
  "file": "base64-строка"
}

Плюсы:
- удобно для обычных данных

Минусы:
- плохо подходит для файлов
- base64 увеличивает размер примерно на 33%
- не нужен, если хочется просто загрузить видео

2. multipart/form-data
- Самый частый способ для файлов.
- В одном запросе отправляются и поля, и сам файл.

Пример curl:
curl -X POST "http://127.0.0.1:8000/v1/video/" \
  -F "title=Урок 1" \
  -F "author=Нана" \
  -F "course=Курс 1" \
  -F "file=@./video.mp4"

Плюсы:
- просто
- стандартно для файлов
- удобно для FastAPI и браузеров

Минусы:
- запрос не JSON, а форма
- если нужен только файл без метаданных, это может быть избыточно

3. raw binary
- В теле запроса только байты файла.
- Метаданные идут отдельно, например в URL или заголовках.

Пример:
curl -X PUT "http://127.0.0.1:8000/v1/video/1/file" \
  --data-binary @video.mp4

Плюсы:
- подходит, если отправляешь только файл

Минусы:
- сложнее передавать title, author, course
- обычно нужен отдельный запрос на создание записи
- не такой удобный вариант для типичного CRUD API

Вывод:
- JSON - для обычных данных.
- multipart/form-data - для данных и файла.
- raw binary - только файл.

Краткий вывод по FastAPI
- Form(...) означает, что поле приходит из multipart/form-data, а не из JSON.
- File(...) означает, что это поле содержит файл.
- Для учебного проекта и обычной загрузки видео multipart/form-data - самый простой и нормальный выбор.
- Клиент API сам отправляет файл, руками на сервере его прикреплять не нужно.

Для текущего проекта лучше использовать multipart/form-data.
Это самый простой и нормальный вариант.
Файл не нужно прикреплять вручную на сервере.
Клиент API сам отправляет файл в запросе.
